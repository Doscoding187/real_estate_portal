# Stakeholder And Domain Registries

This file classifies the current product surface by stakeholder and domain boundary. Every classification is current-state first; target-state recommendations are called out separately.

## Stakeholder Registry

| Stakeholder | Primary jobs in the platform | Current surfaces | Main data owners | Current maturity | Key ambiguity |
| --- | --- | --- | --- | --- | --- |
| Visitor, buyer, tenant, searcher | Search inventory, inspect property/development pages, save or enquire, request services | Public search routes, property detail, development detail, developer brand profile, services directory, Explore/discovery, favorites, saved searches | `properties`, `developments`, `unitTypes`, `developerBrandProfiles`, `leads`, `favorites`, `savedSearches`, `serviceLeads` | Established public experience | Search cards are blended from multiple sources with different source-of-truth rules. |
| Seller, landlord, private owner | Create or promote a property listing, respond to enquiries | Listing wizard routes and listing engine | `listings`, `listingMedia`, `listingApprovalQueue`, public `properties` mirror | Established for listing lifecycle, less clear as a named stakeholder domain | Owner identity and post-lead CRM are not as strongly separated from agent/agency workflows. |
| Agent | Maintain profile, manage listings, receive enquiries, schedule showings, participate in referrals/distribution | Agent routes, listing engine, agent dashboard, distribution partner/referrer pages | `agents`, `agentCoverageAreas`, `listings`, `leads`, `showings`, `distributionDeals`, `referrals` | Established | Agent profile, user account, agency membership, and distribution identity overlap. |
| Agency admin/team | Onboard agency, manage branding/team/listings/subscriptions | Agency routes, agency admin dashboard, invitations, branding | `agencies`, `agencyBranding`, `agencyAgentMemberships`, `invitations`, `agencySubscriptions`, `agents` | Established | Agency membership is mature, but not generalized to developers or service providers. |
| Property developer | Maintain developer account/profile, create developments, manage unit types, publish developments, handle leads, view funnel/analytics, use distribution | Developer workspace, development wizard, developer brand profile, development detail, developer lead/funnel endpoints, distribution dashboard | `developers`, `developerBrandProfiles`, `developments`, `unitTypes`, `developmentDrafts`, `leads`, `developerSubscriptions`, `distributionPrograms` | Established and central | Developer account, public brand profile, marketing agency identity, and team permissions need clearer boundaries. |
| Marketing agency / brand operator | Publish or operate developer brand/development content on behalf of a brand | Developer wizard identity fields, `developerBrandProfiles.identityType`, super-admin publisher context | `developerBrandProfiles`, `developments.marketingBrandProfileId`, `developments.marketingRole` | Emerging | The role exists in schema and publish payloads, but organizational permission and ownership lifecycle are not fully isolated. |
| Service provider | Register provider profile, list services/locations, receive service leads, publish Explore videos, manage reviews | Services pages, service provider onboarding, service provider dashboard, services engine router | `serviceProviderProfiles`, `serviceProviderServices`, `serviceProviderLocations`, `serviceLeads`, `serviceProviderReviews`, `explorePartners` | Emerging-to-established | Provider identity depends on Explore partner identity, which mixes content partner and service provider concepts. |
| Distribution referrer / partner agent | Access referral-enabled developments, submit referrals/deals, schedule viewings, earn commissions | Distribution partner pages, referral accelerator, partner program terms, submit referral/deal flows | `distributionIdentities`, `distributionAgentAccess`, `distributionAgentTiers`, `distributionDeals`, `distributionViewings`, `distributionCommissionEntries`, `referrals` | Established but feature-gated | The actor can be an agent, a referrer, or both; identity ownership is still domain-local. |
| Distribution manager / platform ops | Configure programs, assign managers, validate viewings, advance deals, manage documents and commissions | Super-admin distribution admin pages, manager invite flow, manager checklist/pipeline pages | `distributionManagerAssignments`, `distributionPrograms`, `distributionDevelopmentAccess`, `distributionDealDocuments`, `distributionViewingValidations`, `distributionDealEvents` | Established but operationally concentrated | Large router has multiple bounded contexts inside one surface. |
| Super admin | Approve, publish, moderate, verify, configure platform/domain settings | Admin dashboard, super-admin publisher, distribution admin, service moderation, developer approval | `users`, audit logs, domain admin tables | Established | Super-admin powers cut across all domains; domain-specific admin contracts should stay explicit. |
| Finance partner / contractor | Potential future platform actor | Economic actor schema only | `economicActors`, `financeProfiles`, `contractorProfiles` | Candidate | The schema exists but does not yet own the product flows. |

## Domain Engine Registry

| Domain engine | Status | Owns | Why this classification | Boundary notes |
| --- | --- | --- | --- | --- |
| Single-Property Listing Engine | Established | `listings`, listing media, approval queue, listing readiness, published `properties` mirror | Has schema, router, lifecycle, approval, media sync, public projection, and contract tests | Do not collapse into Development Listing; the inventory model and authoring lifecycle differ. |
| Development Listing Engine | Established | Developer workspace, development drafts, developments, unit types, publish workflow, public development detail, derived search cards | Has wizard, canonical payload, service layer, public consumers, lead/funnel endpoints, and tests | Needs clearer ownership between `developers` and `developerBrandProfiles`. |
| Public Property Catalog / Search Results | Established public read-model/query capability and public experience | `properties` projection, derived development listing cards, property search service, search filters | Public search blends single-property and development-derived inventory with tests | It is not the write owner for every property-like thing. |
| Discovery / Explore Feed | Emerging-to-established discovery/content capability | Discovery feed, ranking, engagement, Explore content/videos | New `server/domains/discovery` exists with contracts/tests, while legacy Explore services still exist | Needs a migration map between legacy Explore and new Discovery domain. |
| Agency And Agent Operations | Established mixed stakeholder operations capability | Agency onboarding, branding, memberships, agent profiles, agent dashboard workflows | Separate agency and agent schemas/routers/services have strong evidence, but the workspace label is not one monolithic backend engine | Shares lead/showing concepts with generic lead system and listing engine. |
| Service Provider Engine | Emerging-to-established | Service provider profile, services/locations, service directory, service leads, reviews, Explore videos | Distinct schema/router/service/UI/tests exist | Identity should be separated from or deliberately tied to Explore partner identity. |
| Distribution And Referral Engine | Established but boundary-risky | Distribution programs, access, deals, viewings, documents, commissions, referral affordability | Deep schema, router, services, tests, and UI pages | Large router contains admin, manager, agent/referrer, developer, and public onboarding contexts. |
| Demand Lead Engine | Emerging | Demand campaigns, demand leads, matching, assignment, unmatched leads | Schema and router exist | Overlaps with generic lead intake and marketing/campaign language. |
| Brand Profile / Developer Marketplace Identity | Emerging shared-domain hybrid | Developer brand profiles, brand lead capture, public developer pages | Strongly used by Development Listing and lead routing | Should be treated as public commercial identity, not only developer account data. |
| Billing / Entitlements | Shared capability, fragmented | Plans, subscriptions, developer limits, agency subscriptions, service subscriptions, partner subscriptions | Multiple working billing surfaces exist | Needs a shared entitlement interface before schema unification. |
| Reviews / Trust | Thin shared capability | Service provider reviews, stub public reviews router | Service reviews exist; generic reviews route returns empty list | Do not promise a cross-domain review engine yet. |

## Shared Capability Registry

| Capability | Current owner or implementation | Maturity | Duplication risk | Target posture |
| --- | --- | --- | --- | --- |
| Identity and auth | `users`, auth router/service, role guards | Established | Role values are shared, but actor profiles are domain-specific | Keep `users` as login/account root; expose typed actor identity adapters per domain. |
| Organizations and membership | Agencies have the strongest model; developers and services have lighter models | Partial | Developer teams and service provider teams are not equivalent to agency membership | Do not generalize until one developer/team contract is proven. |
| Roles and permissions | tRPC procedure types, route guards, user roles, super-admin procedures | Partial | Super-admin and domain roles cut across many routers | Add domain-level permission decision helpers before broad RBAC refactor. |
| Contact identity | Users, agents, agencies, developer brand profiles, service provider profiles, lead records | Fragmented | Public contact identity is repeated and sometimes copied into leads | Define a contact identity projection and make lead capture store source references. |
| Media | Upload router/S3-style services, listing media, property images, development JSON media, service Explore videos | Partial shared infrastructure | Attachments are domain-owned even when upload plumbing is shared | Keep storage shared, ownership and lifecycle domain-local. |
| Notifications and email | `notifications`, email service, domain-specific notification tables/usages | Partial | Domain-specific notifications can bypass shared preferences | Add event naming and recipient identity contracts. |
| Analytics and event transport | `analyticsEvents`, `locationAnalyticsEvents`, AgentOS events, Explore analytics, listing analytics | Fragmented but useful | Metrics are reported differently per domain | Standardize event envelope first, not reports. |
| Billing and entitlements | Billing schema/router plus developer, agency, service, partner subscriptions | Fragmented | Subscription state appears in several tables and on `users` | Create entitlement read adapters per stakeholder before schema consolidation. |
| Reviews and trust | Service provider reviews plus stub generic reviews router | Thin | Public UI could imply cross-domain review support that does not exist | Build a shared review target contract before reuse. |
| Lead intake and routing | `leadsRouter`, `publicLeadCaptureService`, brand lead service, service leads, demand leads, distribution deals | Fragmented by design and history | A single "lead" term means different lifecycles | Treat shared intake as capture and attribution; keep CRM pipelines domain-specific. |
| Search indexing/projections | `properties`, derived development listings, location pages, discovery candidates | Established read concerns with write ambiguity | Public catalog can be mistaken for source of truth | Make source metadata mandatory in public result contracts. |
